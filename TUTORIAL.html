1. What Does This App Do?

Imagine you're a media analyst at a research firm. Every day, dozens — sometimes hundreds — of news articles mention pharmaceutical companies like Pfizer, Eli Lilly, or Johnson & Johnson. Your job is to read each article, decide whether it's relevant, record the company mentioned, note the tone (positive, negative, neutral), and tag what topic it covers (earnings, lawsuits, new drugs, etc.).

Doing that by hand for hundreds of articles is exhausting. findMentions automates the tedious parts:

  • It opens each article URL automatically.
  • It finds and highlights every mention of the company you're looking for.
  • It figures out when the article was published.
  • It lets you click KEEP or DELETE, choose a sentiment, pick a topic, and move straight to the next article — all without touching a URL or a spreadsheet manually.
  • When you're done, it produces a finished spreadsheet with everything filled in.
The big picture analogy
Think of the extension as a research assistant who sits beside you, opens each article in a browser tab, uses a yellow highlighter pen to mark all company mentions, then waits while you make a judgment call before flipping to the next article — all while keeping a running tally in a spreadsheet.

Who Uses It?

Media analysts and researchers at Gravity Research who review news coverage of pharmaceutical and medical-device companies. The app is designed specifically for the pharma/medtech space, with built-in support for company name variations ("GSK" vs "GlaxoSmithKline") and false-positive suppression ("Lilly" should not match "Lilly Pulitzer").

2. What Is a Chrome Extension?

Before we can understand how the code works, we need to understand what a Chrome Extension is — and what makes it different from an ordinary website.

A website vs. an extension

When you visit a website, the website's code lives on a server somewhere and runs inside your browser tab. It can only affect that one tab, and it can't access your other tabs or your browser settings.

A Chrome Extension is a small program that's installed inside Chrome itself, not just in a tab. Because of that special position, it can do things a website can't:

A regular website can...A Chrome Extension can also...
Show you content in a tabOpen a side panel attached to the whole browser window
Read and modify its own pageRead and modify any page you open (with your permission)
Store data in a cookieStore data in Chrome's built-in storage (survives tab close)
Make network requestsIntercept and modify network requests from any tab
Run only when a tab is openRun a background process that wakes on events

Permissions: asking for trust

Because extensions are powerful, Chrome requires them to declare upfront exactly what they can do. This declaration lives in a file called manifest.json. When you install an extension, Chrome shows you the permissions list so you can decide whether to trust it.

Key concept: permissions
A permission is a specific capability that the extension is allowed to use. For example, the "tabs" permission lets an extension know which tabs are open. Without it, the extension is blind to them. findMentions must declare the permissions it needs to scan web pages, open a side panel, and store your review progress.

3. Architecture: Three Worlds Inside One Extension

findMentions is made of three separate programs that run in three completely isolated environments. They can only talk to each other through a specific messaging system — like three employees in different rooms who can only communicate by passing notes through a slot in the wall.

Click on each box in the diagram below to learn what that part does:

background.js — The Gatekeeper

This is a tiny program that runs in the background of Chrome. In Manifest V3, it's called a service worker, which means it wakes up only when something happens (like the user clicking the extension icon), does its job, and goes back to sleep. It can't hold onto data between wakeups.

What it does in findMentions:
  • When you click the extension icon → opens the side panel
  • Ensures the side panel is enabled on every new tab
  • Logs startup/install events
Why it doesn't do more: The side panel (sidebar.js) can talk directly to the content script — the background doesn't need to relay messages.
sidebar.js — The Brain

This is the main program. It lives in the side panel — the UI you see on the right side of your browser. Unlike the background script, it stays alive as long as the panel is open.

What it does:
  • Loads and parses your CSV files
  • Builds the regex patterns for company name matching
  • Opens article URLs in browser tabs
  • Tells the content script to scan the page
  • Receives results and updates the UI
  • Saves your progress to chrome.storage.local
  • Writes the output CSV
content.js — The DOM Worker

This script is injected into every article page you open. It runs inside the article's web page — but in an isolated bubble, so it can't accidentally break the page or access the page's own JavaScript variables.

What it does:
  • Scans the page's text for company name matches
  • Wraps matches in yellow highlight spans
  • Handles Shift+click to remove a highlight
  • Extracts article body text
  • Detects the article's publication date
The "note through the wall" rule: content.js can only talk to the side panel by calling chrome.runtime.sendMessage(). It cannot call most Chrome APIs directly.
The Article Page — Raw Material

This is the actual news article opened in a Chrome tab. The browser builds a DOM (Document Object Model) from the page's HTML — essentially a tree of every paragraph, heading, image, and link on the page.

content.js walks this tree to find company name mentions and inserts highlight spans into it. The highlights appear visually on the page because they're real DOM modifications — the same as if you opened the browser's developer tools and manually edited the HTML.
chrome.storage.local — The Filing Cabinet

Chrome provides extensions with their own persistent key-value storage. Think of it as a filing cabinet that survives browser restarts.

findMentions stores: your CSV data, current position in the review, company variations map, topic hierarchy, and column name mappings — so if you close the extension and come back tomorrow, you pick up exactly where you left off.

Limitation: Each value has a size limit. Article body text (which can be long) is stripped before saving to avoid hitting the quota.

4. The Manifest: The Extension's ID Card

Every Chrome Extension must include a file called manifest.json. Think of it as the extension's official ID card — it tells Chrome who the extension is, what it wants to do, and what special abilities it needs permission to use.

Analogy: a building access badge
When a new contractor arrives at a building, they're given a badge that says "this person is allowed to enter rooms A, C, and F — not B or D." The manifest is that badge. Chrome reads it when the extension is installed and enforces those access rules forever.

Why does it need "scripting" AND "content_scripts"?

The content_scripts block is a declarative injection: Chrome automatically injects content.js into every page when it loads. The "scripting" permission enables a programmatic injection: sidebar.js can manually inject content.js into a tab that was already open before the extension loaded — a fallback safety net. Both are needed.

5. The Side Panel: sidebar.js

sidebar.js is the longest and most complex file. It's organized as a single JavaScript class — think of a class as a blueprint for an object that has both data (like a spreadsheet's current row number) and abilities (like "load a CSV file" or "move to the next article").

Analogy: the class as a filing system
The CSVReviewer class is like a filing system sitting on your desk. It holds folders for your CSV data, your current position, your company name lists, and your topic categories. It also has a set of procedures — labeled steps you can follow to do things like "load a new file" or "decide on an article."

CSV parsing: reading a spreadsheet

A CSV file (Comma-Separated Values) is a plain text file where each row of a spreadsheet is one line, and cells are separated by commas. For example:

example.csv
company,link,date
Pfizer,https://example.com/article1,2026-04-22
Eli Lilly,https://example.com/article2,2026-04-23

The extension has its own hand-written CSV parser that handles a tricky edge case: what if a cell's text contains a comma? (Article body text often does.) The solution is quoting: if a field contains a comma, wrap the whole field in double quotes. The parser correctly handles this, plus escaped quotes ("" inside a quoted field means a literal ").

📖 Deep dive: how the RFC 4180 CSV parser works

The parser reads the file one character at a time. It tracks whether it's inside a quoted field or not. If it encounters a " character when not in a quoted field, it enters quote mode and collects everything until the closing " — even if that spans multiple lines. This is how multi-paragraph article body text can be stored in a single cell.

sidebar.js — parseCSV() simplified
if (text[i] === '"') {
  // Quoted field: read until the next " that isn't doubled
  let field = '';
  i++;  // skip the opening quote
  while (i < n) {
    if (text[i] === '"' && text[i+1] === '"') {
      field += '"'; i += 2;  // "" → one literal "  
    } else if (text[i] === '"') {
      i++; break;           // closing quote → end of field
    } else {
      field += text[i++];   // regular character → add to field
    }
  }
  row.push(field);
}

Company variations: the lookup tables

The extension needs two lookup tables built from the company variations CSV. Here's what happens when it reads a row like Eli Lilly,!Lilly Endowment:

  • Column 1 (Eli Lilly) → the canonical (official) company name
  • Column 2 starts with ! → this is an exclusion phrase, not a search term
  • The phrase after ! (Lilly Endowment) goes into exclusionsMap["Eli Lilly"]
Result after loading companies_to_find.csv (Eli Lilly rows)
// variationsMap: what to search FOR
variationsMap["Eli Lilly"] = ["Eli Lilly", "Lilly"]

// exclusionsMap: what to NEVER match
exclusionsMap["Eli Lilly"] = ["Lilly Endowment", "Lilly Foundation", "Lilly school"]

Generating a regex pattern

This is the heart of the search engine. A regular expression (regex) is a special text pattern that describes what to search for — like a smarter, more powerful version of Ctrl+F. Instead of searching for one exact phrase, a regex can express rules like "match any of these five spellings, but only as a whole word, and not if followed by these words."

Analogy: a regex is like a postal sorting rule
Imagine you're sorting mail and you write a rule: "Deliver to the Lilly family — but NOT if the envelope says 'Lilly Endowment' or 'Lilly Foundation', because those are different organizations." That's exactly what the exclusion regex does.

📖 How generateCompanyRegexPattern() works — step by step

The function builds the pattern in five steps. Here's the algorithm traced through the Eli Lilly example:

sidebar.js — generateCompanyRegexPattern("Eli Lilly")
// Step 1: Collect all search terms
allTerms = ["Eli Lilly", "Lilly"]  // company + variations

// Step 2: Escape special characters for regex
// "Johnson & Johnson" → "Johnson \& Johnson" (& is special in some regex flavors)
cleanedTerms = ["Eli Lilly", "Lilly"]

// Step 3: Build exclusion suffixes
// "Lilly Endowment" starts with "Lilly" → suffix is "endowment"
// "Lilly Foundation" → suffix is "foundation"
// "Lilly school"     → suffix is "school"
suffixes = { "endowment", "foundation", "school" }

// Step 4: Wrap suffixes in a "negative lookahead"
// (?!\s+(?:...)) means "not followed by whitespace then any of these"
negativeLookahead = "(?!\\s+(?:endowment|foundation|school))"

// Step 5: Assemble the final pattern
// \b = word boundary (only match whole words)
pattern = "\\b(?:Eli Lilly|Lilly)(?!\\s+(?:endowment|foundation|school))\\b"
flags   = "gi"  // g = find ALL matches; i = case-insensitive

The word boundary (\b) is a magic position in the regex that matches where a word starts or ends — between a letter/digit and a space or punctuation. This prevents "Lilly" from matching "Lillyhammer" or "illicitly".

The negative lookahead ((?!...)) says "match here ONLY IF what follows does NOT match this pattern." It fires immediately after the match, before the final word boundary. So "Lilly Endowment" is tested as: does "Lilly" match? Yes. Does what follows match \s+(?:endowment|foundation|school)? Yes → reject the whole match.

The article review flow

When the analyst clicks "Start Processing," the extension finds the first article that hasn't been reviewed yet and begins the review cycle. Here's the sequence:

1
Find the first row in csvData without a KEEP/DELETE label.
↓
2
Check if there's a preloaded background tab for this URL. If yes, activate it instantly. If no, navigate the current tab to the article URL.
↓
3
Wait for the page to finish loading (Chrome fires a "complete" event). Register the listener before navigating to avoid missing fast-loading pages.
↓
4
Ping the content script to confirm it's running. If not, inject it manually.
↓
5
Send the company's regex pattern to the content script → it scans the page, highlights matches, and reports back how many were found.
↓
6
Simultaneously, ask the content script to detect the article's publication date.
↓
7
Show the review panel. The analyst reads the article, navigates highlights, and clicks KEEP or DELETE.
↓
8
On KEEP: auto-extract article body text if a content column exists. Write the decision + all fields to csvData. Save state. Close the tab. Move to article N+1.

Auto-preloading: staying ahead

After each article loads, the extension immediately opens the next 1–3 article URLs in background tabs (tabs the analyst can't see yet). By the time the analyst finishes reviewing the current article, the next ones are already loaded. This hides the page-load wait time.

The preloaded tabs are stored in a Map (a data structure that connects keys to values, like a dictionary). When the extension needs the next article, it checks this map first.

Multi-company rows

Sometimes an article mentions multiple companies. If the analyst checks more than one company in the checklist and clicks KEEP, the extension creates duplicate rows in the spreadsheet — one for each additional company — so each company gets its own review record. The duplicates are inserted right after the current row and automatically skipped during the next review cycle (they're already tagged).

The BigQuery Workflow

The extension has two completely independent ways to load its article work queue: from a CSV file on your computer, or directly from BigQuery — Google's cloud database service. Both paths lead to the same article review experience; what changes is where the data comes from and where decisions are recorded. Neither workflow is "the primary one" — which you use depends on how your team works.

Analogy: desk filing cabinet vs shared cloud cabinet
CSV mode is like a paper filing cabinet sitting on your own desk. You pull out a folder, work through it, write your notes, and put it back. Nobody else sees your notes until you email them a copy. BigQuery mode is a shared filing cabinet in the cloud. Every note you write is immediately visible to the whole team. There is no "your copy" — just one live record everyone reads and writes to simultaneously.

Authentication: connecting to Google

BigQuery is a secured Google Cloud service. The extension needs an access token — a temporary credential issued by Google after sign-in — to read and write data. This credential-exchange process is called OAuth 2.0. Think of it as a visitor badge: Google issues one after verifying your identity, and it's valid for one hour.

Here is exactly what happens when the analyst clicks "Connect to Google":

1
Build the sign-in URL. bqConnect() assembles a URL pointing to Google's OAuth sign-in page. It includes the OAuth Client ID, the scope being requested (https://www.googleapis.com/auth/bigquery), and a redirect URI that Chrome owns — unique to this specific extension installation.
↓
2
Open the Google sign-in popup. chrome.identity.launchWebAuthFlow({ interactive: true }) opens a popup managed entirely by Chrome. The analyst signs in with their Google account inside it. The extension never sees the password.
↓
3
Google sends the token back. After a successful sign-in, Google redirects the browser to the extension's unique redirect URI (e.g. https://abc123.chromiumapp.org/). The access token and its expiry time are embedded in the URL hash — the text after the #.
↓
4
Parse and store the token. Chrome passes the full redirect URL back to the extension. The code splits the hash on &, reads access_token and expires_in, and stores them in memory. The stored expiry is set to 60 seconds before the actual expiry as a safety buffer — the extension re-auths slightly early rather than sending a request with an already-expired token.
sidebar.js — bqConnect() (condensed)
const redirectUri = `https://${chrome.runtime.id}.chromiumapp.org/`;
const scope = encodeURIComponent('https://www.googleapis.com/auth/bigquery');
const authUrl = 'https://accounts.google.com/o/oauth2/v2/auth'
  + `?client_id=${encodeURIComponent(bqConfig.clientId)}`
  + `&redirect_uri=${encodeURIComponent(redirectUri)}&response_type=token&scope=${scope}`;

const responseUrl = await chrome.identity.launchWebAuthFlow({ url: authUrl, interactive: true });
const hash = new URL(responseUrl).hash.substring(1);
const params = Object.fromEntries(hash.split('&').map(p => p.split('=')));

this.bqToken       = params.access_token;
this.bqTokenExpiry = Date.now() + (parseInt(params.expires_in, 10) - 60) * 1000;

Fetching the work queue from BigQuery

After signing in, the analyst clicks "Load Data." The extension runs two SQL queries against BigQuery simultaneously:

  1. A data query that fetches up to 500 unreviewed articles.
  2. A count query that tells the analyst how many unreviewed articles remain in total (so the status bar can show "Loaded 500 of 1,847 total — reload for more").

The core logic uses a LEFT JOIN. Think of two guest lists: everyone invited to a conference (processed_serp_results) and everyone already checked in (validated_results). The LEFT JOIN finds every person on the first list who does not appear on the second — the unchecked-in attendees who still need to be reviewed.

sidebar.js — bqLoadData() main SQL query
SELECT * EXCEPT(rn)
FROM (
  SELECT p.*,
    -- Number rows within each (URL, company) group to eliminate duplicates
    ROW_NUMBER() OVER (PARTITION BY p.link, p.company ORDER BY p.link) AS rn
  FROM `sri-benchmarking-databases.coverage_collector.processed_serp_results` p
  LEFT JOIN `sri-benchmarking-databases.coverage_collector.validated_results` v
    ON LOWER(TRIM(p.link))    = LOWER(TRIM(v.link))
   AND (p.company IS NULL
        OR LOWER(TRIM(p.company)) = LOWER(TRIM(v.company)))
  WHERE v.link IS NULL                      -- rows with no match = not yet validated
    AND (p.date IS NULL OR p.date >= '2026-04-01')
)
WHERE rn = 1                              -- drop duplicates: keep first per group
ORDER BY (company IS NULL OR TRIM(company) = '') ASC, company, outlet
LIMIT 500

Key clauses explained:

  • LOWER(TRIM(...)) — normalizes case and whitespace before comparing. "Pfizer" and " pfizer " are treated as the same.
  • WHERE v.link IS NULL — after a LEFT JOIN, rows that found no matching partner in the right table have NULL in all right-side columns. This clause filters to only those unmatched rows — the ones not yet validated.
  • ROW_NUMBER() OVER (PARTITION BY p.link, p.company) — a window function that assigns a sequence number within each (URL, company) group. WHERE rn = 1 keeps only the first occurrence of any duplicate in the source table.
  • LIMIT 500 — the per-session batch cap. If more unreviewed articles remain, the status message prompts the analyst to reload for the next batch.

Writing decisions back to BigQuery

When the analyst clicks KEEP or DELETE, writeValidationToBigQuery() fires immediately. A parameterized SQL INSERT statement runs as a BigQuery job. The decision lands in validated_results and is visible to the whole team right away — no download, no upload, no merge.

Parameterized queries protect against errors from special characters. Instead of injecting values directly into the SQL string, named placeholders like @p_company are used, and the actual values travel in a separate queryParameters array. BigQuery combines them safely server-side.

sidebar.js — writeValidationToBigQuery() (key lines)
// All source columns + decision + timestamp
const row = { ...sourceFields, decision, validated_at: new Date().toISOString() };
const cols = Object.keys(row);

// BQ param names must be [a-zA-Z][a-zA-Z0-9_]* — prefix with p_ to be safe
const pname = c => `p_${c.replace(/[^a-zA-Z0-9_]/g, '_')}`;
const colList   = cols.map(c => `\`${c}\``).join(', ');
const paramList = cols.map(c => `@${pname(c)}`).join(', ');
const sql = `INSERT INTO \`${projectId}.${datasetId}.validated_results\` (${colList}) VALUES (${paramList})`;

// Each column becomes one parameter descriptor
const queryParameters = cols.map(c => ({
  name: pname(c),
  parameterType: { type: c === 'validated_at' ? 'TIMESTAMP' : 'STRING' },
  parameterValue: { value: row[c] === null ? null : String(row[c]) }
}));

Two columns distinguish validated_results from the source table:

  • decision (STRING) — the analyst's verdict: "KEEP" or "DELETE".
  • validated_at (TIMESTAMP) — the exact moment the decision was made, in ISO 8601 format (e.g. 2026-04-22T14:30:00.000Z).

Auto-creating the output table

The first time "Load Data" is clicked, bqEnsureValidatedTable() runs a safety check. If validated_results doesn't exist yet, the extension creates it automatically — you never have to set it up manually in BigQuery Console.

1
Check if the table exists. A GET request asks the BigQuery API for the validated_results table metadata. If it succeeds, the table exists — stop here, nothing to do.
↓
2
If not found (HTTP 404): read the source schema. Fetch the column definitions (field names and data types) from processed_serp_results.
↓
3
Append the two validation columns. Add decision (STRING) and validated_at (TIMESTAMP) to the end of the schema array.
↓
4
Create the table. POST the complete schema to BigQuery. The table is ready for writes. All future sessions skip straight to step 1.

Token expiry and re-authentication

Google OAuth tokens expire after one hour. The extension handles re-authentication automatically so the analyst doesn't need to think about it. Before every API call, bqGetToken() checks whether the current token is still valid:

sidebar.js — bqGetToken()
async bqGetToken() {
  if (this.bqToken && this.bqTokenExpiry && Date.now() < this.bqTokenExpiry) {
    return this.bqToken;  // still valid — use it as-is
  }
  // Expired or never set — silently re-authenticate
  await this.bqConnect();
  return this.bqToken;
}

When the token expires mid-session, bqConnect() runs again. Because the analyst already granted the extension permission, Chrome often completes the re-auth silently — no popup appears. A popup reappears only if the analyst's underlying Google session has also expired (uncommon during a normal work session).

6. The Content Script: content.js

content.js is injected into every article page. It's where the actual page reading and highlighting happens. The whole script lives inside a class called PageSearcher.

The message dispatch table

The content script doesn't start doing anything on its own — it waits for instructions from the side panel. Those instructions arrive as messages (think of them as phone calls). The script has a dispatcher that reads the action field of each message and routes it to the right function:

content.js — message dispatch
switch (request.action) {
  case 'searchAndHighlightRegex':  // Highlight one specific company
    result = await searchAndHighlightRegex(pattern, flags, terms, company);
    break;
  case 'scanForCompanies':          // Find ALL companies on the page
    result = await scanForCompanies(companies);
    break;
  case 'navigateMatch':             // Jump to prev/next highlight
    result = navigateMatch(direction);
    break;
  case 'scanForDate':               // Auto-detect publish date
    result = findDate();
    break;
  case 'extractArticleText':        // Extract clean article body
    result = extractArticleText();
    break;
  case 'clearHighlights':           // Remove all highlights
    clearHighlights();
    break;
  case 'ping':                      // "Are you there?" health check
    sendResponse({ ok: true });
    break;
}

Walking the DOM with TreeWalker

Before we explain the search functions, we need to understand how the content script reads the page.

A web page's content is organized in a tree structure: the page is divided into sections, sections contain paragraphs, paragraphs contain sentences. At the very bottom of the tree are text nodes — the actual words you read.

Analogy: the postal worker
Imagine a city where streets contain buildings, buildings contain apartments, and apartments contain mailboxes. A TreeWalker is like a postal worker who visits every mailbox in the city, in order, starting from the first address and going all the way to the last — never skipping one, never going backward. The content script uses a TreeWalker to visit every text node on the page.
content.js — TreeWalker pattern
// Create a walker that visits only text nodes
const walker = document.createTreeWalker(
  document.body,           // Start at the top of the page
  NodeFilter.SHOW_TEXT,   // Only visit text nodes (not tags)
  {
    acceptNode: (node) => {
      // Skip script and style blocks — those aren't visible text
      if (node.parentNode.tagName === 'SCRIPT') return NodeFilter.FILTER_REJECT;
      if (node.parentNode.tagName === 'STYLE')  return NodeFilter.FILTER_REJECT;
      // Skip empty/whitespace-only nodes
      if (!node.textContent.trim()) return NodeFilter.FILTER_SKIP;
      return NodeFilter.FILTER_ACCEPT;  // OK to scan this one
    }
  }
);

let textNode;
while ((textNode = walker.nextNode())) {
  // textNode.textContent is the actual text in this part of the page
  // Run the regex against it here...
}

highlightMatches: the group-based DOM algorithm

Once we have a list of matches (each match knows: which text node it's in, where it starts, where it ends), we need to visually highlight them on the page. This means wrapping the matched text in a <span> element with a yellow background class.

This sounds simple, but there's a nasty trap: what if two matches are in the same text node?

The crash that happened
The old code processed matches in reverse order (last match first). For a node with matches at positions 5 and 20: process position 20 first — that splits the text node and removes the original from the DOM. Then try to process position 5 — but the original node is gone. Result: TypeError: Cannot read properties of null. The extension crashed.

The fix: group all matches that share the same text node, then process each group in a single forward pass using the original node as a fixed anchor until every match has been processed.

📖 The full highlightMatches() source code — annotated
content.js — highlightMatches()
highlightMatches() {
  // ── Step 1: Group consecutive same-node matches ──────────────────
  // TreeWalker visits in document order, so same-node matches are adjacent.
  const groups = [];
  for (let i = 0; i < this.matches.length; i++) {
    const last = groups[groups.length - 1];
    if (last && last.node === this.matches[i].node) {
      last.indices.push(i);  // same node as previous match → add to group
    } else {
      groups.push({ node: this.matches[i].node, indices: [i] }); // new group
    }
  }

  // ── Step 2: Process each group in one forward pass ───────────────
  for (const { node, indices } of groups) {
    const text   = node.textContent;  // full text of the original node
    const parent = node.parentNode;   // the element containing this text
    if (!parent) continue;           // safety check

    let cursor = 0;  // our position in the text string
    for (const idx of indices) {
      const { startIndex, endIndex } = this.matches[idx];

      // Insert text BEFORE this match (the gap since the last highlight)
      if (startIndex > cursor) {
        parent.insertBefore(
          document.createTextNode(text.substring(cursor, startIndex)),
          node   ← node is still in DOM here, acts as anchor
        );
      }

      // Create the yellow highlight span
      const span = document.createElement('span');
      span.className = 'csv-reviewer-highlight';
      span.textContent = text.substring(startIndex, endIndex);
      parent.insertBefore(span, node);  ← insert BEFORE anchor

      this.matches[idx].element = span;  // save reference for navigation
      cursor = endIndex;
    }

    // Insert remaining text AFTER the last match
    if (cursor < text.length) {
      parent.insertBefore(document.createTextNode(text.substring(cursor)), node);
    }

    // NOW remove the original node — all its content is in new nodes
    parent.removeChild(node);
  }
}

Shift+click to remove a highlight

Each highlight span has a special click handler. When the analyst holds the Shift key and clicks a highlight, that single highlight instance is removed — without affecting other instances of the same company name on the page.

This is useful for false positives: if "Lilly" correctly matched 5 times but one of those is actually "Lilly Pulitzer" (which slipped through), Shift+clicking that one removes it from the count without clearing all highlights.

Why Shift+click instead of just click?
Highlight spans are often inside hyperlinks (<a> tags). A plain click on a link triggers navigation — the page would reload and all highlights would be lost. Using Shift+click avoids the ambiguity: the extension calls e.preventDefault() to block navigation AND e.stopPropagation() to stop the event from reaching the link handler.

After removing a highlight, the content script notifies the side panel:

extractArticleText: getting the article body

When the analyst clicks KEEP, the extension tries to extract the clean article body text and save it in the spreadsheet. The content script uses two strategies, tried in order:

Strategy 1 — known article containers: Many news sites use standard CSS class names or HTML attributes for their article body. The script tries a prioritized list:

content.js — article container selectors (in priority order)
[
  'article [class*="body"]',     // e.g. <div class="article-body">
  '[class*="article-content"]',   // common pattern
  '[itemprop="articleBody"]',      // Schema.org structured data
  '[data-testid="article-body"]', // React-based news sites
  'article',                       // HTML5 semantic element
  'main'                           // Last resort: the main content area
]

Strategy 2 — paragraph density scoring: If no standard container is found, the script scores every <div>, <section>, and <main> element by adding up the lengths of its <p> paragraphs. The element with the highest score wins — it probably contains the most article text.

Once the container is identified, only meaningful text tags are collected: <p>, headings (<h1>–<h6>), list items, and blockquotes. Navigation menus, social sharing buttons, comment sections, and advertisements are filtered out.

findDate: four-tier date detection

The content script tries to find the article's publication date using four increasingly unreliable methods:

TierMethodWhy it's reliable / unreliable
1 (best)Meta tags like article:published_timeSite explicitly declares the date in machine-readable format
2HTML <time> elements with datetime attributeSemantic HTML — well-structured sites use this
3JSON-LD structured data (datePublished)SEO-conscious sites embed this for Google
4 (least reliable)Regex on the first 3,000 characters of page textWorks for "Jan 30, 2023" style dates but can match copyright years

The side panel only accepts dates from the last two years — older dates are treated as stale and ignored (they're likely copyright years or boilerplate).

7. The Background Service Worker: background.js

background.js is the shortest file in the project — intentionally. In Manifest V3, background scripts are service workers: they can't hold onto data or keep running indefinitely. They start, do a job, and stop.

findMentions needs the background script for exactly one reason: only the background script can receive the event that fires when the user clicks the extension icon, and only the background script can call chrome.sidePanel.open().

background.js — the complete file
// Open the side panel when the extension icon is clicked
chrome.action.onClicked.addListener(async (tab) => {
  await chrome.sidePanel.open({ windowId: tab.windowId });
});

// Enable the side panel on every tab that becomes active
chrome.tabs.onActivated.addListener(async (activeInfo) => {
  await chrome.sidePanel.setOptions({ tabId: activeInfo.tabId, enabled: true });
});

Everything else — state management, CSV parsing, messaging the content script — lives in sidebar.js, which stays alive as long as the panel is open.

10. Key Design Decisions and Gotchas

Why regex instead of simple text search

The simplest approach to finding "Pfizer" in an article would be page.includes("Pfizer"). This fails for several reasons:

  • Case sensitivity: "PFIZER" and "pfizer" wouldn't match.
  • Substring matches: "Pfizer" would match inside a longer word.
  • Multiple spellings: You'd need a separate search for each variation.
  • False positives: "Lilly" would match "Lilly Pulitzer" with no way to suppress it.

Regex with the i flag solves case sensitivity. Word boundaries (\b) prevent substring matches. Alternation ((?:Eli Lilly|Lilly)) matches all spellings in one pass. Negative lookahead prevents false positives. One regex rule replaces what would otherwise be many lines of if/then logic.

The DOM crash and how the group-based highlighter fixes it

See Section 6.4 above. The key principle: never remove or re-parent a DOM node that another piece of code still holds a reference to. The old reverse-order approach removed the original text node too early. The new forward-pass approach uses the original node as a read-only anchor until all new nodes are inserted, then removes it last.

The race condition in tab loading

When the extension calls chrome.tabs.update(tabId, { url }) to navigate a tab, the tab starts loading. Chrome fires a chrome.tabs.onUpdated event when loading completes. If the extension registers the listener after navigation starts, a fast-loading page might fire "complete" before the listener is attached — and the extension would wait forever.

The fix: register the listener before calling chrome.tabs.update(). The listener checks that the tab ID matches before acting, so it ignores events from other tabs. A 90-second safety timeout unblocks the UI if the page never fires "complete."

The [auto] / [setup] / [skip] commit markers

The repository has a post-commit hook (a script that runs automatically after every git commit). It generates an HTML writeup explaining what changed in that commit. To prevent infinite recursion — the writeup commit triggering another writeup — commit messages that contain [auto], [setup], or [skip] are ignored by the hook.

Rule of thumb: any commit you create manually doesn't need a marker. If a script or automated process creates a commit, include one of these markers so the hook doesn't cascade.

Storage quota and article text

chrome.storage.local has a per-item size limit. Article body text (sometimes thousands of words) can easily exceed this. The extension strips the content column before saving state (saveState()) and re-scrapes it on the next KEEP. This means: if you clear all data without downloading the CSV first, article text is lost — but other fields (KEEP/DELETE, sentiment, topic) are preserved.

BigQuery mode vs CSV mode

The extension supports two data sources: a CSV file you upload, or a BigQuery table in Google Cloud. Both modes use identical review UI and csvData array. The difference:

  • CSV mode: Load a file → review → download a new file. All data stays local.
  • BigQuery mode: Authenticate with Google → query unreviewed rows → review → each KEEP/DELETE is written back to BigQuery in real time via the REST API.

In BQ mode, saveState() does not persist csvData to chrome.storage because the data lives in BigQuery and should be re-fetched (to get the freshest unreviewed rows) on next session.
